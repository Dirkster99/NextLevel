Features

This list describes main features as of Version 2.00, Build 30.

  • Every procedure or trigger (short: PROC) is listed with a:
    • a list of calling procedures,
    • a maximum callgraph,
    • and a list of accessed tables or views.
  • Names of tables and procedures are highlighted and linked within the HTML source code. Click on the name of a table or procedure and you should be transported to a page summarizing the technical details of that database object.
  • An INSERT, DELETE, or UPDATE statement on a table can cause a trigger to be invoked, if the targeted table has a trigger installed. This can be seen as an indirect extension of a callgraph, because a procedure called STORE_ITEM could insert a record in a table called LINE_ITEM. LINE_ITEM in turn may have an INSERT trigger installed, which will be invoked before the procedure STORE_ITEM is continued.

  • There is a summary page for each table. This summary page contains a list of procs that implement a SELECT, INSERT, DELETE, or UPDATE statement on a given table. Click on the statement and you should be transported to the first occurrence in the target procedure, -click on the procedure name and you should be transported to the PROC... well, you get the idea, don't you!?

  • Installed triggers for every table are shown on the index HTML page and the table summary page. Click on the table name to get to the summary page of each trigger.

  • The parser is able to recognize INSERTED and DELETED tables in triggers. Access statements are counted as accessing the base table. If we have for example a trigger T_SALES_TRIG_U on a SALES table and the trigger implements an SQL statement like SELECT * FROM inserted, then this is correctly recognized as a select statement on the SALES tables.

  • Each page generated by Nextlevel has a menu in the right upper corner. In this menu there is an entry called "Unreferenced" . This entry leads to a summary page that is similar to the main index page, however, all procedures, triggers, tables, and views listed on the "Unreferenced" page are listed, because there is no evidence for their usage within the database. No evidence of usage means that Nextlevel did not encounter one SQL statement that makes use of this object. The database object could therefore, be obsolete, or be used via dynamic SQL, or be used by an external source (e.g. a client program or from an external database or server).
  • In the menu on the right upper corner of each page, there is an entry called "Connections" , which leads to a summary page for database objects that access objects outside the current database.

  • In SQL, it is possible to declare a cursor in one part of the SQL code and use that cursor (a few 100 lines) later in the same proc. Nextlevel links the declaration statement with all other statements referring to that cursor. It is thus, very easy to jump from the declaration to the usage of a cursor (or back) without having to read all intermediate lines.

  • NextLevel integrates with the HTML Help Workshop, which can be downloaded for free from Microsoft's homepage (I hope to integrate other help systems soon). The Windows Help System has the advantage that all output files are kept in one physical file, using a minimum of space through compression, but, at the same time, being very accessible through full text search features.

  • Resolving SQL database views is supported through a list of all recognized definitions on the right side of the database index page.

    The definition page for a view looks very similar to that of an SQL table, except that views are shown with their SQL source code, which is displayed and highlighted in the definition page. The view definition page shows seperate tables for database objects that access the view (procedures, views, or triggers) and database objects that are accessed by the view (tables or views). Also, each column and their datatype is shown to make accessing particular columns and their names easy as a pie ;)

Known issues with NextLevel:

  • Parsing SQL 92 style joins (LEFT OUTER, INNER etc.) is now supported, however a SQL 92 FROM clause may not be parsed correctly, if there are sub-selects within the FROM clause.
    • Symptoms: Some table dependencies following a sub-select in an SQL-92 FROM clause may not be recognized by the parser. You may notice missing SELECT dependencies, as a result.
