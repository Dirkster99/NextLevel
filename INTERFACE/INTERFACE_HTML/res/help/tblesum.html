Table Summary Page

Understanding the role of a table in a database involves the knowledge of each program accessing it, -its size, and the design of each column and other details. This and more valuable information is summarized on one page (for each table) labeled with an icon "" of a table.

The data shown below in Figure 4 gives an example of a summary page for a table named COST_TYPES. This tables has 2 columns called "TYPE_ID" and "DESCRIPTIONvc" of the data type int and varchar, as we can see in the table labeled "Table Definition". We can also see just above that table how much space this table costs in terms of data, index, and total size (if this information was imported via XML).

The list in the table labeled "Objects Accessing Table" shows the names of each procedure or trigger that implement an SQL statement on this table. The left column shows the objects, click on them and you should see the summary page for that proc, while the right side show each class of implementing statement (SELECT, INSERT etc.), just click on the statement and you will be transported to its first occurrence in the proc.

We can see in the figure above that COST_TYPES is accessed from within the database by no more 5 triggers, which select information from the table. This knowledge by itself can lead us to conclude that data in COST_TYPES is either static or fed into the database by external sources (e.g. client program).

The table COST_TYPES has 2 triggers as we can see in the 2 entries labeled with a proc icon "". Click on each name and you will see the trigger summary page.
